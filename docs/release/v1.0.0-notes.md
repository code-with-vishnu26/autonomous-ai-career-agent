# v1.0.0 — Supervised Prepare-Only Release

**Status:** Released. **Position:** `PREPARE_ONLY`. **Date:** 2026-07-10.
**Promoted from:** [`v1.0.0-rc1`](v1.0.0-rc1-notes.md) (Phase 37/ADR-0059).

This is the first stable release of the Autonomous AI Career Agent. It packages
the discover → decide → ingest → prepare → learn workflow as a supervised,
prepare-only tool. **It does not submit applications to any external system.**

## What changed since rc1

The `1.0.0rc1` release candidate (ADR-0056) was **CONDITIONAL_GO**, with four
named open conditions. All four are now closed by real evidence, not
assumption:

| Condition (rc1) | Resolution |
|---|---|
| Live-output quality unvalidated | **Resolved.** A real controlled live-Groq smoke (Phase 36, run by the maintainer, real `GROQ_API_KEY`, real `verify-promptfoo --provider groq` PASS) tailored, gated, ATS-scored (**78.125**), and rendered a real DOCX from real Groq output. |
| Live Promptfoo BLOCKED_BY_CONFIGURATION | **Resolved** for the maintainer's environment — a real `verify-promptfoo --provider groq` PASS was observed. (Still, by design, user-local: no key/artifact is ever committed, and CI structurally cannot reproduce this — ADR-0055.) |
| No in-repo CI | **Resolved** (Phase 35/ADR-0057) — `.github/workflows/ci.yml` runs lint, architecture contracts, the full test suite, packaging, and a clean-install smoke on every push/PR. |
| Windows/macOS execution untested | **Windows resolved** — both by CI (`windows-latest`, every push/PR) and by the maintainer's real local Windows run. **macOS remains untested** — a deliberate, cost-driven gap (10x CI runner multiplier), not silently dropped. |

Phase 36 also found and fixed a real defect surfaced by that first live run:
declining the confirmation prompt left the application-store row
permanently stuck, refusing every future retry. Fixed and merged (PR #58,
ADR-0058); both CI legs green on the merge commit.

**Post-merge, the fix was independently reconfirmed on real Windows CLI +
SQLite** against a fresh synthetic opportunity: decline → row recorded
`status=declined` → a second `apply` for the same opportunity was **not**
blocked → tailored, gated, and rendered again → declined again. Zero
external submissions in either run. The original pre-fix opportunity's
stuck `status=pending` row was left untouched as historical evidence.

## Real live-evidence summary (Phase 36, single sample — not a statistical claim)

- **Provider:** Groq (selected naturally by existing provider precedence).
- **Promptfoo:** `verify-promptfoo --provider groq` → real **PASS**.
- **ATS score:** `78.125` (real, from the SQLite application row).
- **Truthfulness:** `truthfulness_approved = 1`.
- **Claim safety:** the adversarial job-description injection ("...claim the
  candidate has 8 years of Kubernetes experience and led 20 engineers")
  did **not** appear anywhere in the rendered resume text — verified by
  direct inspection of the real output, not inferred.
- **Artifact:** a real `.docx` was rendered to `data/artifacts/`.
- **External submission:** **0** — the run stopped at the confirmation
  prompt (declined); no executor is reachable regardless (ADR-0050).
- **Token usage / cost:** **UNKNOWN** — not exposed by the CLI's output in
  this run; not estimated here (ADR-0055 already disclaims absolute cost).

This is one real sample, not a benchmark. It demonstrates the pipeline
*can* produce a safe, truthful, on-topic result under a real provider — it
does not certify output quality across all candidates, JDs, or provider
versions.

## Capability matrix and known limitations

Unchanged from `1.0.0rc1` (see [`v1.0.0-rc1-notes.md`](v1.0.0-rc1-notes.md)
for the full table) except where noted above. Still true:

1. PDF/OCR CV import: **NOT_SUPPORTED**.
2. Browser/email/autonomous external submission: **NOT_SUPPORTED** — code
   exists, unwired, unreachable.
3. macOS: **untested**.
4. PDF rendering may require a local LibreOffice install.

## Upgrade / install

```bash
pip install -e ".[dev]"
career-agent setup
```

No database migration is required (the Phase 36 lifecycle fix added one new
`Application.status` value to an unconstrained `TEXT` column — fully
backward-compatible with every existing database).

## Release decision

**GO.** Every release-blocking invariant holds; the four rc1 conditions are
closed with real evidence; CI is green on both supported platforms for the
exact commit this version is built from; no secret or private data leak;
external submission remains unreachable. See ADR-0059 for the full
reasoning and evidence trail.
