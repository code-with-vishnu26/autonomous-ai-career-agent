# v1.1.0 — Portability, Packaging & Onboarding Hardening

**Status:** Promotion prepared — the git tag and GitHub Release are a
separate, manual maintainer step (never performed by the agent) and remain
pending owner authorization. **Position:** `PREPARE_ONLY` (unchanged).
**Date:** 2026-07-10. **Promoted from:** [`v1.0.0`](v1.0.0-notes.md)
(Phase 43–44 / ADR-0062).

This is a minor release on top of the stable v1.0.0. It keeps the
supervised, prepare-only discover → decide → ingest → prepare → learn
workflow exactly as it was — **it still does not submit applications to any
external system** — and adds one backward-compatible configuration surface
plus a set of packaging, onboarding, and documentation-accuracy fixes.

## Why this is a MINOR bump (SemVer)

The only runtime source change since v1.0.0 is a new, optional,
`.env`-overridable `Settings` field. Per SemVer 2.0.0 that is new
backward-compatible functionality → MINOR. See
[ADR-0062](../adr/0062-v1-1-readiness-audit-and-version-decision.md) for the
full audit and reasoning (why not v1.0.1, why not "remain unreleased").

## What changed since v1.0.0 (Phases 38–42)

| Area | Change | Phase / ADR |
|---|---|---|
| **Runtime config** | New `Settings.promptfoo_results_dir` (CWD-relative default `promptfoo/results`, `.env`-overridable), replacing the old `__file__`/install-tree-relative path resolution. Fixes a real bug: the old default only resolved correctly for an editable install — a wheel/non-editable `pip install` pointed it uselessly into `site-packages`. | Phase 40 / ADR-0060 |
| **Packaging** | A real sdist leak was found and fixed (untracked, non-`.gitignore`d local directories were swept into the sdist); `scripts/verify_release_artifacts.py` gained a **positive top-level allowlist** so an unanticipated directory fails closed. The `Development Status` classifier was corrected `2 - Pre-Alpha → 5 - Production/Stable`. | Phase 41 / ADR-0061 |
| **Onboarding docs** | Quick Start split into separate Bash and PowerShell blocks; a correct camelCase profile example that loads through the real CLI loader; a Promptfoo pointer. A later fix corrected a README line that falsely framed its example as the literal scaffold output. | Phases 39, 42 |
| **Secret hygiene** | Local Phase-36 smoke-evidence filenames and agent/editor session-state dirs (`.claude/`, `.import_linter_cache/`) are now git-ignored by name. | Phases 38, 41 |

## Behavior compatibility

For the **documented, supported** usage (editable install, run from the repo
root), `promptfoo/results` resolves to exactly the same path as before —
**no behavior change.** The change is a strict bugfix for the previously
broken wheel/non-editable case, plus one new optional config knob. **No
dependency versions changed.**

## Live-LLM evidence

**No fresh live-LLM run was required for this release, and none is claimed.**
No LLM-facing code changed since v1.0.0 — the truthfulness gate, both
`ClaimVerifier` implementations, the prompt text, and the promptfoo prompt
version (`truthfulness-gate-v2`) are byte-identical to v1.0.0. The v1.0.0
controlled live-Groq evidence (Phase 36 / ADR-0059) therefore carries
forward unchanged. As with v1.0.0, CI itself makes **no** live LLM call (no
key, egress blocked by design).

## Capability matrix and known limitations

Unchanged from v1.0.0. Still true:

1. PDF/OCR CV import: **NOT_SUPPORTED**.
2. Browser/email/autonomous external submission: **NOT_SUPPORTED** — code
   exists, unwired, unreachable (`executor_available=False`, hardcoded).
3. macOS: **untested** — a deliberate, documented gap.
4. PDF rendering may require a local LibreOffice install.

## Upgrade / install

```bash
git pull
pip install -e ".[dev]"
career-agent setup
```

No database migration is required. No configuration change is required: the
new `promptfoo_results_dir` defaults to the same `promptfoo/results` path
the tool already used for supported usage.

## Release decision

**GO** for v1.1.0 — pending only the maintainer's manual tag/GitHub Release
step, which the agent never performs on its own. Every release-blocking
invariant that held for v1.0.0 still holds (architecture contracts, the
prepare-only execution boundary, the fabrication and Promptfoo gates, no
secret/private-data leak, external submission unreachable); the full test
suite and CI are green on both supported platforms for the commit this
version is built from; and no LLM-facing behavior changed. See ADR-0062 for
the readiness audit, the SemVer reasoning, and the risk register (no P0/P1).

### Maintainer tag/release step (run manually, not by the agent)

After this promotion PR is **merged** and CI is green on the merge commit:

```bash
git fetch origin main
git checkout main && git pull
# annotated tag, on the merge commit:
git tag -a v1.1.0 -m "v1.1.0 — portability, packaging & onboarding hardening"
git push origin v1.1.0
```

Creating the GitHub Release from that tag (optional) is likewise a manual
maintainer action. Do not move, delete, or force-push the immutable `v1.0.0`
tag.
