# ADR-0048: Application-attempt idempotency guard (Phase 22, bounded slice)

- **Status:** Accepted
- **Date:** 2026-07-06
- **References:** [ADR-0014](0014-opportunity-identity-and-deduplication.md)
  (opportunity-level identity, unmodified and untouched by this ADR),
  [ADR-0037](0037-persistence-discover-and-first-profile-writer.md)
  (`SqliteApplicationStore`, extended here with one new read method),
  [ADR-0041](0041-scheduling-gates-and-bounded-auto.md) (`auto`'s
  structurally-cannot-submit guarantee, unchanged)

## Context

Phase 22's brief asked for a broad production-reliability review: failure
taxonomy, retry/backoff, checkpointing, resumability, observability, and
more, across Discovery/Decide/Resume-tailoring/Application/Email/
Persistence/Notifications. The brief itself required a repository-reality
audit before any implementation, and explicitly permitted (Stage 17)
implementing only the smallest, most evidence-backed subset rather than
everything the brief enumerated.

The audit (fresh reads, not memory) found this project's actual current
architecture is: single-process, single-user, at-most-once in-process
event bus (`core/bus.py`, explicitly documented as non-durable and
never a safety mechanism), stdlib `sqlite3` used synchronously per-call
(no cross-request transaction or checkpoint concept needed, since there
is no distributed or concurrent writer), no network-retry logic anywhere
(`retry`/`backoff` search hits are exclusively about the unrelated
ATS-gate content-retailor loop, Phase 10), no `run_id`/`execution_id`/
checkpoint/transaction/rollback/observability-framework concept anywhere
in the codebase, and per-source discovery failures already fail-and-skip
cleanly (`run_discover_command`'s `except Exception: print(...); continue`).
Given the project's own stated real scale ("tens of applications, not
thousands," ADR-0039) and single-user local deployment, none of
Kafka/Temporal/Celery/Redis/Kubernetes-class infrastructure, a formal
checkpoint/resumability journal, or an automatic-retry engine is
justified by anything actually observed in this repository.

One concrete, real gap was found, though, with a genuine safety
consequence: **there is no protection anywhere against re-attempting an
application to the same opportunity across two separate process
invocations.**

- `domain/identity.py`'s `opportunity_id()`/`canonical_fingerprint()`
  (ADR-0014) dedupe *opportunities* at discovery time -- they say nothing
  about whether an opportunity has already been *applied to*.
- `Application.id` is a fresh `uuid.uuid4()` generated by
  `ResumeTailoringPipeline.run()` on every single call
  (`agents/resume/pipeline.py`) -- never derived from `opportunity_id` or
  any other stable input.
- `SqliteApplicationStore.record()`'s `INSERT OR IGNORE INTO applications
  (id, ...)` is keyed on that fresh `id` (`applications.id` is the sole
  `PRIMARY KEY`; there is no uniqueness constraint on `opportunity_id`),
  so it can never actually catch a second attempt against the same
  opportunity -- every call inserts a new row.
- `cli.py` never queried `application_store` before invoking the
  tailoring pipeline (confirmed by grep: exactly two references to
  `application_store.` existed before this change, both `.record(...)`
  calls -- never a lookup).

Concretely: running `career-agent apply` twice against the same
opportunity file, or letting `career-agent auto` re-discover and re-rank
an opportunity across two cron ticks before its handoff file is acted on,
would silently tailor, gate, and (if a human confirms both) submit twice
-- a genuine duplicate real-world side effect, not a cosmetic one.

## Decision

Add a minimal, read-before-write idempotency guard, nothing more:

1. `SqliteApplicationStore.prior_attempt_status(opportunity_id) -> str |
   None` (`storage/sqlite.py`): returns the most recent recorded
   application status for `opportunity_id` whose status is **not**
   `"rejected"`, or `None` if none exists. `"rejected"` is excluded
   deliberately: a truthfulness-gate rejection (ADR-0003) produced no
   external side effect at all, so a fresh attempt after the user fixes
   their profile or the drafted content is a legitimate, safe retry --
   blocking it would only add friction with no safety benefit.
2. `cli._apply_pipeline` (behind `career-agent apply`) calls this check
   first, before `ResumeTailoringPipeline.run()` is ever invoked. If a
   prior non-rejected attempt exists, it refuses outright: prints the
   opportunity id and the prior status, explains the duplicate-submission
   risk, and exits 1 -- never a silent skip, and never an automatic
   override.
3. `cli.run_auto_command` (behind `career-agent auto`) runs the same
   check per ranked opportunity, inside its existing loop. A blocked
   opportunity is skipped with a printed explanation and the loop
   continues to the next one -- appropriate for an unattended, multi-
   opportunity, cron-invoked pass, where one already-attempted
   opportunity must not halt the whole run.
4. In both cases, the check only runs when `application_store` is
   supplied (matching the existing optional-injection pattern both
   functions already used for `.record()`) -- callers that don't pass a
   store see no behavior change.

This directly answers the brief's own research questions about
duplicate-side-effect risk: yes, one opportunity could previously be
applied to twice through different invocations (RQ2/RQ17); yes, current
opportunity-level fingerprinting protects discovery-time deduplication
only, never the separate application-attempt layer (RQ18). After this
change, a second attempt is a refused, visible, human-resolved decision,
never a silent duplicate.

## What was deliberately *not* built, and why

Consistent with the brief's own evidence-gated instruction not to
implement everything it enumerates:

- **No formal execution state machine.** The existing `Application.status`
  literal (`"pending" | "paused_for_human" | "submitted" | "failed" |
  "rejected"`) and this one new read are sufficient to answer "has this
  opportunity already been attempted, in a way that matters." Building a
  richer `(S, E, δ)` transition model would describe transitions this
  codebase doesn't yet drive through any separate orchestrator -- there is
  no evidence a bigger model is needed yet.
- **No automatic retry or backoff.** No network-retry logic exists
  anywhere in this codebase today, and none is added here: every
  discovery-source failure already fails-and-skips per source
  (`run_discover_command`), and this ADR's own guard's entire point is
  that a repeat *application* attempt must be a human decision, never
  automatic. Adding retry machinery elsewhere with no corresponding
  observed failure to justify it would be exactly the kind of
  unjustified sophistication the project's stated discipline forbids.
- **No checkpoint/resumability journal.** `career-agent auto`'s own
  handoff-file design (ADR-0026/0041) is already the resumability
  mechanism: a prepared-but-unconfirmed application sits as a durable
  JSON file on disk until the human runs `apply` against it, surviving a
  process crash trivially. A separate journal would duplicate this with
  no new capability.
- **No `SUBMISSION_UNCERTAIN` state.** No code path in this repository
  currently produces genuine submission-outcome ambiguity (a timeout
  *after* a real send call, for instance) -- `TieredApplicator.submit()`
  and `BrowserApplicator`'s `_finish()` only ever return a definite
  `ApplicationSubmitted` or a definite `ApplicationFailed`/exception; the
  email tier never claims `ApplicationSubmitted` at all (ADR-0021). There
  is nothing to name a new ambiguous state for yet.
- **No new observability framework.** `core/bus.py`'s existing event
  catalog (`ResumeTailored`, `TruthfulnessRejected`, `HumanActionRequired`,
  `ApplicationFailed`, `OutcomeRecorded`, ...) plus `SqliteApplicationStore`'s
  already-durable audit trail are the existing, adequate observability
  surface for this project's actual single-user scale; this ADR adds one
  new read against that same durable store rather than a parallel
  logging/metrics system.
- **No token/cost accounting changes.** Out of scope for this slice; no
  new evidence was found this phase that changes the prior "not tracked,
  not fabricated" status quo.

## Consequences

- `storage/sqlite.py`: one new method, `SqliteApplicationStore.
  prior_attempt_status()`. No schema change (reads the existing
  `applications` table by the existing, already-indexed-by-nothing-new
  `opportunity_id` column via a plain `WHERE` clause -- table is small
  enough at this project's real scale that no new index is warranted).
- `cli.py`: `_apply_pipeline` and `run_auto_command` each gained one
  guard clause, run only when `application_store` is supplied. Zero
  change to `ResumeTailoringPipeline`, the truthfulness gate, the ATS
  gate, `TieredApplicator`/`BrowserApplicator`/`EmailApplicator`,
  confirmation/submission safety, or `auto`'s structurally-cannot-submit
  property.
- New tests: `tests/storage/test_sqlite_store.py` (three cases: no prior
  attempt, a `"rejected"` prior attempt is ignored, a `"submitted"` prior
  attempt is reported), `tests/test_cli_apply.py` (a prior non-rejected
  attempt refuses before tailoring is even attempted, proven by an
  `input_fn` that raises if ever called), `tests/test_cli_auto.py` (a
  ranked pass with one already-attempted and one fresh opportunity
  records exactly the fresh one, skipping the other with a printed
  explanation).
- Zero cost, zero network, zero new dependency.
